Week 10

Timeline: 22/6 → 28/6 (5 working days)

Day 1 - 22/6: Studying the AWS Security Monitoring Architecture

Tasks Completed: I began studying AWS security monitoring and alerting services to understand how automated incident response can be implemented in the cloud. I explored the roles of Amazon GuardDuty, Amazon EventBridge, and Amazon SNS in detecting security threats and triggering automated workflows. I also analyzed the lifecycle of a security finding, from detection to incident creation within the Incident Response Management System (IRMS).

Knowledge Gained: I learned that Amazon GuardDuty continuously analyzes AWS CloudTrail logs, VPC Flow Logs, and DNS logs to identify suspicious activities. When a potential threat is detected, GuardDuty generates a Security Finding that can be processed automatically by other AWS services.

Achievements: I gained a comprehensive understanding of the overall security monitoring architecture and how multiple AWS services collaborate to automate incident detection and response.

Challenges and Lessons Learned: This was my first experience working with a complete cloud security monitoring solution, so I spent considerable time reviewing AWS documentation. I realized that understanding the event flow makes system implementation much easier.

Day 2 - 23/6: Configuring Amazon GuardDuty

Tasks Completed: I enabled Amazon GuardDuty in my AWS account and explored the different categories of Security Findings generated by the service. I performed several simulation exercises to observe how GuardDuty detects suspicious activities and classifies findings according to their severity levels.

Knowledge Gained: I learned that Amazon GuardDuty leverages machine learning and threat intelligence to identify malicious behavior without requiring additional software agents on servers. Each Security Finding contains detailed information that assists security teams in assessing potential risks.

Achievements: I successfully enabled GuardDuty and generated Security Findings for testing the incident response workflow.

Challenges and Lessons Learned: Initially, interpreting the Security Findings was challenging because they contained numerous attributes. After studying the AWS documentation, I gained a better understanding of each field and how to evaluate the associated security risks.

Day 3 - 24/6: Integrating Amazon EventBridge with AWS Lambda

Tasks Completed: I configured Amazon EventBridge to monitor Security Findings generated by Amazon GuardDuty. I then created EventBridge Rules to automatically invoke AWS Lambda whenever new security events were detected. The Lambda function processed each finding and generated a corresponding incident within the IRMS.

Knowledge Gained: I learned that Amazon EventBridge supports event-driven architectures by automatically routing events that match predefined rules to target services such as AWS Lambda. This eliminates the need for manual intervention when processing security events.

Achievements: EventBridge successfully triggered Lambda functions whenever GuardDuty generated new Security Findings, and the event data was correctly processed by the IRMS.

Challenges and Lessons Learned: During configuration, I encountered issues caused by incorrect event filtering rules. After refining the Event Pattern, Lambda successfully received only the intended GuardDuty events.

Day 4 - 25/6: Sending Security Notifications with Amazon SNS

Tasks Completed: I studied Amazon Simple Notification Service (SNS) to deliver notifications whenever new incidents were created. I created an SNS Topic, configured an email subscription, and integrated the notification process with AWS Lambda so that email alerts would be sent automatically after a new incident was successfully generated.

Knowledge Gained: I learned that Amazon SNS follows a Publish/Subscribe messaging model, allowing a single message to be delivered simultaneously to multiple subscribers through email and other supported communication protocols. This makes SNS suitable for real-time security notifications.

Achievements: The system successfully sent email notifications whenever Amazon GuardDuty detected a security event and AWS Lambda created a corresponding incident.

Challenges and Lessons Learned: Email subscriptions must be confirmed before notifications can be delivered successfully. I learned the importance of validating every step in the notification workflow to ensure alerts reach their intended recipients.

Day 5 - 26/6: Testing the Automated Monitoring Workflow and Weekly Review

Tasks Completed: I tested the complete automated workflow, beginning with threat detection by Amazon GuardDuty, followed by event routing through Amazon EventBridge, incident creation by AWS Lambda, and email notification delivery through Amazon SNS. Together with my teammates, I reviewed the workflow, examined CloudWatch logs, and documented any issues that required further improvement.

Knowledge Gained: I gained a deeper understanding of event-driven architectures on AWS and how multiple managed services can be integrated to build an automated incident response system. I also realized that monitoring CloudWatch logs is essential for debugging and validating serverless applications.

Achievements: I completed the implementation of the automated incident creation and notification workflow. The system operated reliably and fulfilled the project’s functional requirements.

Challenges and Lessons Learned: During testing, some events were not processed because the EventBridge Rules were not configured correctly. After reviewing the event patterns and verifying the Lambda execution permissions, the workflow functioned as expected. I learned that when working with serverless architectures, each stage of the processing pipeline should be verified individually to quickly identify and resolve potential issues.

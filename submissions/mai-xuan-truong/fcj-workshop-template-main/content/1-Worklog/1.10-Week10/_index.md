**Timeline:** 22/6 → 26/6 (5 working days)

## Day 1 - 22/6: Topic preparation

**Work completed:** I reviewed possible project directions based on the AWS services learned so far. I prioritized a topic that could include authentication, APIs, serverless compute, database, storage, event-driven automation, deployment, and documentation. This was still topic preparation, not the actual implementation phase.

**Knowledge learned:** A good internship project should have business value and also demonstrate the AWS knowledge learned during the internship.

**Result achieved:** I had clearer criteria to discuss project selection with the team.

**Difficulty and lesson learned:** The scope needed to be realistic for the remaining internship time.

## Day 2 - 23/6: Architecture drafting

**Work completed:** I prepared a rough serverless architecture using CloudFront, API Gateway, Cognito, Lambda, DynamoDB, S3, EventBridge, SNS, CloudWatch, and Secrets Manager. This draft was only for discussion, not the final architecture. I arranged services into layers: frontend delivery, authentication, API layer, compute, data, notification, and monitoring.

**Knowledge learned:** A rough architecture helps the team understand the system before dividing work.

**Result achieved:** I had an initial service map based on my AWS learning.

**Difficulty and lesson learned:** Draft diagrams must be updated after the team confirms real requirements.

## Day 3 - 24/6: Proposal outline

**Work completed:** I outlined the proposal with context, problem statement, objectives, expected features, service mapping, technical risks, and deliverables. I wrote it as a team project while keeping my personal focus on AWS infrastructure, deployment, testing, and documentation. This prepared the team for the official topic discussion.

**Knowledge learned:** A proposal should explain the problem, solution, and scope, not only list AWS services.

**Result achieved:** I had a proposal structure for the team to review and improve.

**Difficulty and lesson learned:** The proposal should not promise more features than the team can implement.

## Day 4 - 25/6: Workshop structure planning

**Work completed:** I planned the workshop structure: introduction, environment setup, infrastructure configuration, application development, deployment/testing, frontend integration, and cleanup. I also noted that sidebar numbering should be clear and child pages should match the Hugo structure. This would reduce confusion once the project work began.

**Knowledge learned:** A workshop should follow the order in which readers perform the implementation, not only group content by service.

**Result achieved:** I had an initial structure for the Workshop section of the report.

**Difficulty and lesson learned:** If numbering is inconsistent early, sidebar and internal links become time-consuming to fix later.

## Day 5 - 26/6: Pre-project review

**Work completed:** I reviewed the areas where I could contribute once the project started: AWS infrastructure, AWS SAM, Cognito, API Gateway, Lambda, DynamoDB integration, EventBridge, SNS, CloudFront, deployment, testing, and documentation. I also checked my security notes to avoid placing credentials in source code or the report.

**Knowledge learned:** Preparing my personal role helped avoid duplicated work and unrealistic ownership in a five-member team.

**Result achieved:** I was better prepared for the IRMS start with technical and documentation checklists.

**Difficulty and lesson learned:** The worklog should stay honest: earlier weeks were learning and preparation, while IRMS implementation starts in the week including July 1.
